▶ FASE ACTUALARQUITECTURA BASE
Multi-tenant + Multi-channel + Onboarding
La arquitectura que hace que SiuChat sea un producto real para múltiples empresas
6–8 semanas
0 / 22 tareas
▾
Por qué esta fase es la más crítica del proyecto
Todo lo que viene después — roles, tiempo real, multimedia — se construye encima de esta arquitectura. Si multi-tenant está mal diseñado, cada feature siguiente acumula deuda técnica. Esta fase se diseña en detalle antes de tocar una sola línea de código.
Entorno de desarrollo local
✓
Setup local: PostgreSQL en Docker + FastAPI con uvicorn --reload
✓
Configurar ngrok para apuntar el webhook de Meta al backend local durante desarrollo
✓
Validar que el flujo actual (pre multi-tenant) funciona 100% en local antes de refactorizar
Schema multi-tenant — nuevas tablas
✓
Diseñar y validar en chat: tabla tenants (id, name, plan enum, status enum, timestamps)
✓
Diseñar y validar en chat: tabla users — schema only, sin flow de login (FK a tenant, zoho_user_id nullable, role enum)
✓
Diseñar y validar en chat: tabla channels (phone_number_id unique global, business_account_id, token, display_phone_number, created_by)
✓
Diseñar y validar en chat: tabla contacts por tenant (UNIQUE tenant_id + phone_number, email y notes nullable)
✓
Diseñar y validar en chat: tabla zoho_connections (tenant_id UNIQUE, org_id UNIQUE global, region, token_expires_at)
✓
Modificar conversations: agregar tenant_id, channel_id, contact_id (FK), status enum, last_message_at; eliminar contact_number/contact_name/zoho_contact_id (migran a contacts)
✓
Modificar messages: agregar tenant_id denormalizado
✓
Migración Alembic única y atómica, probada con upgrade + downgrade + upgrade en local
✓
Seed script scripts/seed_initial_tenant.py — crea tenant "Dev Tech Py" + user admin + channel + zoho_connection desde env vars
Integración Zoho por tenant
✓
OAuth flow de Zoho por tenant — cada empresa conecta su propio Zoho (implementación directa, sin MCP)
✓
Refactorizar services/zoho.py para usar credenciales del tenant desde BD, no del .env global
✓
Cache de access_token usando token_expires_at — refresh solo si venció o está por vencer (<5 min)
Meta Embedded Signup — onboarding de canales
✓
Diseñar en chat el flujo completo: login FB → selección negocio → selección/creación cuenta WA → callback → guardar en BD

Endpoint callback que recibe el token y datos del canal y los guarda asociados al tenant

ACCESO Y CONTROL+ FRONTEND
Frontend SPA + Login con Zoho + Roles + Asignación
Migración a SPA, autenticación real, múltiples usuarios por organización con permisos
6–8 semanas
0 / 18 tareas
▾
Por qué se migra el frontend acá y no antes
El vanilla JS del MVP funcionó para Fase 4, pero con login, sesiones, rutas protegidas y múltiples vistas, se vuelve inmantenible rápido. Fase 5 es el momento natural para pasar a SPA porque el frontend ya tiene que manejar autenticación y estado complejo. Se hace antes de agregar tiempo real (Fase 6), para no tener que reescribir la lógica de WebSockets dos veces.

✓

✓
Decidir deploy: FastAPI sirve el build, o Vercel/Netlify con subdominio app.siuchat.com
Encriptación de tokens
✓
Agregar ENCRYPTION_KEY a env vars y generar clave con Fernet
✓
Encriptar channels.token, zoho_connections.access_token y refresh_token a nivel aplicación
✓
Migración de datos: encriptar tokens existentes en la BD actual

✓

✓
Manejo de sesiones — JWT o session token, definir en chat antes de implementar
✓
Proteger el panel — solo usuarios autenticados del tenant correcto pueden acceder
Roles y permisos
✓

✓

✓

Propietario de chat y asignación
✓
Campo assigned_to en conversations (FK → users)
✓
Asignación manual: Admin/Supervisor puede asignar cualquier chat a cualquier agente
✓
Regla de asignación equitativa: nuevo mensaje sin asignar → se asigna al agente con menos chats activos
✓
Auditoría básica: registrar quién respondió qué mensaje y cuándo
Output que demuestra que terminaste
Un Admin crea usuarios en su organización. Cada agente hace login con su cuenta Zoho y ve solo sus chats asignados. El supervisor ve todos. Los nuevos chats se asignan automáticamente de forma equitativa.

CHAT EN TIEMPO REAL
Estados de chat + Mensajes en tiempo real + WebSockets
De un panel que recarga a una experiencia de chat real
4–5 semanas
0 / 12 tareas
▾
Por qué va en este orden
Tiempo real requiere que los usuarios estén identificados (Fase 5) para saber a qué canal de WebSocket conectarlos. Los estados de chat con ventana de 24hs son fundamentales para WhatsApp Business — sin esto el panel muestra información incorrecta a los agentes.
Estados de chat y ventana de 24hs
✓
Diseñar en chat: estados de conversación — abierta / esperando / resuelta / expirada
✓
Lógica de ventana 24hs: si el cliente no escribió en 24hs, solo se puede iniciar conversación con plantilla
✓
Indicador visual de ventana en el panel — el agente sabe si puede responder libremente o no
✓
Job/tarea periódica que actualiza estados expirados automáticamente
Estados de mensajes
✓
Diseñar en chat: recibir webhooks de status de Meta (sent / delivered / read)
✓
Guardar status en tabla messages y mostrarlo en el panel (✓ enviado, ✓✓ entregado, ✓✓ leído en azul)
Tiempo real con WebSockets
✓
Diseñar en chat la arquitectura WebSocket: canales por tenant, reconexión automática
✓
Cuando llega un mensaje nuevo → push via WebSocket al panel del agente correspondiente
✓
Actualización de estado de mensaje en tiempo real (leído, entregado)
✓
Indicador de conexión en el panel — el agente sabe si está conectado al WebSocket
✓
Búsqueda de conversaciones por número, nombre o texto de mensaje
Protección contra bursts
✓
Rate limiting en el webhook — limitar requests por phone_number_id por segundo para proteger contra bursts de Meta (movido desde Fase 8)
Output que demuestra que terminaste
El agente tiene el panel abierto. Un cliente manda un mensaje. Sin recargar la página, el mensaje aparece en tiempo real. El panel muestra si el mensaje fue entregado y leído. El agente sabe si puede responder libremente o necesita una plantilla.

CHAT AVANZADO
Responder mensaje + Ficha de contacto + Plantillas
Las features que convierten el panel en una herramienta de trabajo real
3–4 semanas
0 / 9 tareas
▾
Por qué en este orden
Las plantillas dependen de la lógica de ventana 24hs (Fase 6) — si la ventana está cerrada, el agente necesita plantillas para iniciar. La ficha de contacto se enriquece con datos de Zoho CRM que ya tenés sincronizados. Responder mensaje es UX pura que mejora dramáticamente la experiencia del agente.
Responder mensaje
✓
Diseñar en chat: cómo se guarda la referencia al mensaje original en la DB y cómo se muestra en el panel
✓
UI: click derecho o botón en burbuja → "Responder" → aparece preview del mensaje original sobre el input
✓
Enviar la referencia a la API de Meta para que WhatsApp lo muestre como reply
Ficha de contacto
✓
Panel lateral con datos del contacto: nombre, número, email, empresa (desde Zoho CRM)
✓
Mostrar historial de conversaciones anteriores del mismo contacto
✓
Link directo al contacto en Zoho CRM (ya existe, integrar en la ficha)
Plantillas de Meta
✓
Diseñar en chat: sincronización de plantillas aprobadas desde la cuenta de Meta del tenant
✓
Selector de plantillas en el panel — disponible siempre, obligatorio cuando la ventana 24hs está cerrada
✓
Rellenar variables de la plantilla antes de enviar (nombre del cliente, número de orden, etc.)
Output que demuestra que terminaste
El agente puede responder a un mensaje específico como en WhatsApp. Ve la ficha completa del contacto con sus datos de Zoho. Si la ventana de 24hs está cerrada, selecciona una plantilla aprobada y la envía con las variables completadas.

Pulido del producto — de funcional a placentero de usar

▾
Por qué va al final del ciclo de chat
El multimedia y las notificaciones mejoran la experiencia pero no cambian la arquitectura. Van después de que toda la lógica de negocio esté estable. La carga progresiva también presupone que hay suficientes mensajes en producción para que sea necesaria.

✓

✓
Enviar multimedia: upload desde el panel → enviar via API de Meta
✓
Previsualización en el chat: imagen inline, audio con player, documento con nombre y tamaño
Carga progresiva de mensajes
✓
Diseñar en chat: paginación del GET /conversations/{id}/messages — cargar los últimos N al abrir
✓
Al hacer scroll hacia arriba → cargar bloque anterior sin perder la posición actual
Notificaciones push
✓
Diseñar en chat: Web Push API o servicio externo (OneSignal/Firebase) — evaluar opciones
✓
Notificación cuando llega un mensaje nuevo al agente (aunque no tenga el panel abierto)
✓
Notificación configurable por usuario — el agente puede silenciar
Output que demuestra que terminaste
El agente recibe una notificación push en su celular cuando llega un mensaje. Abre el panel, ve imágenes y archivos inline. En conversaciones largas, el scroll carga mensajes anteriores sin perder el contexto.

ESCALA Y NEGOCIO
Sync Zoho CRM + Auditoría + Suscripciones
Sincronización avanzada, métricas para supervisores, y modelo de cobro
4–5 semanas
0 / 9 tareas
▾
Por qué es la última fase del ciclo principal
La sincronización avanzada con Zoho (owner changes) requiere webhooks de Zoho que a su vez requieren una configuración estable por tenant. Las suscripciones van al final porque primero necesitás clientes usando el producto para saber qué planes tienen sentido.
Sincronización avanzada con Zoho CRM
✓
Diseñar en chat: webhook de Zoho CRM → si el propietario de un Lead/Contacto cambia en Zoho, actualizar assigned_to en SiuChat
✓
Y al revés: si el agente cambia en SiuChat, actualizar el propietario en Zoho CRM
✓
Manejo de conflictos: definir cuál sistema tiene prioridad cuando ambos cambian al mismo tiempo
Auditoría y métricas
✓
Dashboard para supervisores: chats por agente, tiempo promedio de respuesta, chats resueltos
✓
Log de acciones: quién asignó, reasignó, resolvió cada chat
✓
Exportación básica de historial de conversaciones (CSV o PDF)
Suscripciones
✓
Diseñar en chat los planes: Free / Pro / Business — límites por conversaciones, agentes, canales
✓
Integrar Stripe o MercadoPago para cobro recurrente
✓
Límites automáticos por plan — cortar o degradar cuando supera el tier
Output que demuestra que terminaste
Cuando el dueño de un lead cambia en Zoho, SiuChat reasigna el chat automáticamente. El supervisor ve métricas de su equipo. Los tenants pagan por plan y el sistema aplica límites automáticamente. SiuChat es un negocio.
